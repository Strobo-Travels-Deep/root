# Logbook — 2026-04-13 — Engine consistency audit vs Hasse 2024

**Context.** The Fig 6b back-action panel was reproduced only at ~50 %
amplitude *and* with no ϕ-dependence; Fig 8b contrast decay is absent.
User asked for a full code consistency audit of
[scripts/stroboscopic_sweep.py](../../scripts/stroboscopic_sweep.py)
against the Hasse paper Hamiltonian (Appendix C, Eq. C1) and protocol
(Appendix D, Fig 3a sequence). This entry lists every discrepancy I
could identify by reading the engine end-to-end.

Reference: [refs/Hasse2024_PRA_109_053105.pdf](../../refs/Hasse2024_PRA_109_053105.pdf).

-----

## 1. Hasse's reference physics

**Hamiltonian (Appendix C, Eq. C1):**
```
H_TI = ℏ ω_z σ_z / 2  +  ℏ ω_m a† a  +  ℏ Ω/2 [ C(η,a,a†)† σ_-  +  C(η,a,a†) σ_+ ]
```
with `C(η, a, a†) = exp[ iη (a† + a) ]`. Three terms: spin, motional,
spin–motion coupling.

**Protocol (Fig 3a, Appendix D):**
```
Initialise → D(α) → MW π/2 (sync) → AC π/2 train(ϕ)  with  N_S = 30 flashes,
                                                            δt ≈ 100 ns each,
                                                            cycle T_m ≈ 770 ns,
                                                            per-pulse phase set by ϕ
                                                                            → Detect σ_z
```
Fig 6b's ϕ-dependence is generated by the **per-pulse phase progression
set on the AC AOMs**, not by a basis-rotation post-process.

-----

## 2. Engine implementation

**Per-pulse Hamiltonian** ([build_hamiltonian](../../scripts/stroboscopic_sweep.py#L188)):
```python
H[n, n]            = -delta/2                  # |↓⟩|n⟩ block
H[nmax + n, ...]   = +delta/2                  # |↑⟩|n⟩ block
H[:nmax, nmax:]    = (omega_r/2) * C           # ↓⟨↑| block = σ_-  → coupling C·σ_-
H[nmax:, :nmax]    = (omega_r/2) * Cdag        # ↑⟨↓| block = σ_+  → coupling C†·σ_+
```
No `ω_m a†a` term. Coupling order `(C, C†)` × `(σ_-, σ_+)`.

**Inter-pulse propagator**
([run_single](../../scripts/stroboscopic_sweep.py#L353)):
- `t_sep_factor == 1.0` → `Ufree = None`, applied as identity.
- `t_sep_factor != 1.0` → `Ufree[n,n] = exp(-i n · 2π · t_sep_factor)`,
  i.e. exp(−i ω_m a†a · T_pulse) with ω_m·T_m = 2π built in implicitly.

**Initial state** ([prepare_motional](../../scripts/stroboscopic_sweep.py#L142),
[tensor_spin_motion](../../scripts/stroboscopic_sweep.py#L174)):
- Motional: `D(α) |0⟩` with α = |α| exp(iφ_α). ✓
- Spin: `theta_deg=0` ⇒ pure `|↓⟩`. **No MW π/2 sync pulse.**

## 3. Discrepancies (in order of impact on Fig 6b / Fig 8b)

### D1 — Missing ω_m a†a in per-pulse H   ⚑ load-bearing

Hasse Eq. C1 contains `ω_m a†a`. Engine drops it.

The engine's per-pulse propagator is `U = exp(-i δt · (Ω/2)(C σ_- + C†
σ_+))`. The lab-frame Hamiltonian during a pulse is, in the rotating
frame at the laser frequency:
```
H_rot = -δ/2 σ_z  +  ω_m a† a  +  Ω/2 [C† σ_- + C σ_+]
```
The `ω_m a†a` term causes the motional state to rotate by ω_m·δt during
each pulse. With ω_m/(2π) = 1.3 MHz and δt = 100 ns (Hasse) this is
0.82 rad ≈ 47°; with the engine's effective δt ≈ 0.26 (engine units, see
D5) and ω_m = 1.3 it is 0.34 rad ≈ 19°. Neither is "small".

This rotation is what drives Hasse's contrast loss in Fig 8b: at large
|α| the wave packet tangentially traverses a fraction of λ_AC during δt,
the time-averaged AC kick averages out, and the transferred Bloch
amplitude shrinks. The engine, having dropped this rotation, produces a
clean π/2 train at every |α| → flat C(|α|) ≈ 0.92 (our WP-V Fig 8b).

**Fix.** Either (a) add `ω_m a†a ⊗ I_2` to the per-pulse H and let
`expm(-i H δt)` do the work, or (b) move into the interaction picture
w.r.t. `ω_m a†a` and replace `C` with the time-averaged
`C̄ = (1/δt) ∫₀^δt exp[iη(a e^{-iω_m t} + a† e^{iω_m t})] dt` analytically.
Option (a) is cleaner and is what Hasse's QuTip simulation does
(Appendix C: "We numerically analyse the relevant dynamics of Eq. (C1)
via evaluations with QUTIP").

### D2 — No per-pulse analysis-phase parameter on the AC train   ⚑ load-bearing

Hasse Appendix D: "we iteratively readjust the phase of the DDS
controlling the AOMs, incorporating calculated delays". Each pulse k of
the train carries an AC phase ϕ_k that the protocol controls (the
"analysis phase").

In the engine there is no AC-phase argument on `build_hamiltonian` or
`run_single`. The coupling operator is fixed at `C = exp(iη(a + a†))`
with no `e^{iϕ_k}` factor. The only way to access ϕ in the engine's
output is the post-hoc basis-rotation surrogate

```
σ_z(ϕ) = ⟨σ_x⟩ cos ϕ + ⟨σ_y⟩ sin ϕ              (WP-V kickoff)
```

This surrogate is *exact* for the spin readout when the train has zero
AC phase and an analysis π/2 with phase ϕ is applied at the end. It is
*wrong* for any per-pulse phase progression — including the actual Hasse
protocol where ϕ enters every pulse of the 30-flash train. In particular:

- Spin observables `σ_z(ϕ, ϑ₀)` map identically (basis rotation is the
  right surrogate for the σ_z heatmap in Fig 6a, which is why Fig 6a
  passes).
- **Back-action `δ⟨n⟩(ϕ, ϑ₀)` does NOT map** through basis rotation.
  Each AC flash with phase ϕ_k pushes the motional state in a
  direction set by ϕ_k; the cumulative ϕ-dependent kick produces the
  Fig 6b ϕ-fringes that the engine cannot generate.

**Fix.** Add `ac_phase_deg` (per-train scalar) or `ac_phase_schedule`
(per-pulse vector) parameter and dress the coupling block as
`exp(iϕ_k) · C` in `[:nmax, nmax:]` and `exp(-iϕ_k) · C†` in
`[nmax:, :nmax]` per pulse. With this parameter exposed, the WP-V Fig 6b
should then show the four-quadrant ϕ-dependence Hasse plots.

### D3 — Missing MW π/2 synchronisation pulse before AC train

Hasse sequence: `Init → D(α) → MW π/2 → AC π/2(ϕ) → Detect`. Engine
starts in `|↓⟩` (theta_deg = 0 default) and applies the AC train
directly. The MW π/2 puts the spin at the equator before AC, so that
the AC train rotates a *coherence*, not a *population*.

Effect on observables:
- σ_z magnitude: in Hasse's protocol σ_z swings ±1; in the engine's
  protocol the train rotates from |↓⟩ by π/2 toward equator, giving σ_z
  ≈ 0 ± fringe-amplitude (we see +0.128 mean with native readout).
- |C| from (σ_x, σ_y): preserved up to a basis rotation set by which
  axis the MW π/2 chose; magnitude unchanged.

**Fix.** Single-line patch in [tensor_spin_motion](../../scripts/stroboscopic_sweep.py#L174):
expose `mw_phase_deg` and a `mw_pi2=True` flag, or simply default
`theta_deg=90` (puts spin on +x at start) for Hasse-protocol runs. Not
an engine bug per se, but a documented gap between the engine's "AC
train acting on |↓⟩" and Hasse's "AC train acting on a MW-prepared
superposition".

### D4 — Coupling operator placement: (C, C†) vs Hasse (C†, C)

Hasse Eq. C1: `Ω/2 [C† σ_- + C σ_+]`. Engine places `C` with σ_- and
`C†` with σ_+, i.e. the swap. Both forms are Hermitian, so the
spectrum is identical and any |α|-symmetric observable (⟨n⟩, |C|, σ_z²)
is unchanged. The swap is equivalent to η → −η in the kick direction:
asymmetric quantities like `arg C` and the *sign* of phase shifts flip.

Our Fig 8a shows ϕ₀(α=8) = +6.35 rad; Hasse Fig 8a shows
ϕ₀(α≈8) ≈ +2π for the +ve branch. The match suggests this swap is
either a convention difference (Hasse's σ_± labels vs engine's) or that
the relevant observable doesn't see it. Documentation pass needed but
unlikely to be load-bearing.

**Fix.** Either swap the blocks in
[build_hamiltonian:198–200](../../scripts/stroboscopic_sweep.py#L198)
to match Hasse's convention exactly, or add a comment stating the
chosen convention and verifying ⟨X⟩(α=|α|, φ_α=0) > 0 with the engine's
conventions — which is the unambiguous test.

### D5 — Frequency units docstring vs code

[DEFAULTS](../../scripts/stroboscopic_sweep.py#L57) docstring says
`omega_m = 1.3` is "MHz (cyclic, i.e. /(2π))". The code, however,
treats `omega_m`/`omega_r` as angular frequencies in rad / (engine time
unit):

- [run_single:361](../../scripts/stroboscopic_sweep.py#L361):
  `Tm = 2π / omega_m` — only correct if `omega_m` is angular.
- The π/2 budget `N · Ω_eff · dt = π/2`
  ([run_single:360](../../scripts/stroboscopic_sweep.py#L360)) is
  satisfied in **radians** only if `omega_r` is angular.
- [Ufree](../../scripts/stroboscopic_sweep.py#L384):
  `phase_per_n = 2π · t_sep_factor` carries the 2π explicitly, so the
  inter-pulse free evolution agrees with `exp(-i ω_m a†a · t_sep · T_m)`
  for any unit choice (uses dimensionless ratio).

Consequence: **the engine simulates the dimensionless dynamics
correctly** (everything depends on η, Ω/ω_m, N, t_sep_factor — all
dimensionless), but the *absolute* frequency-to-MHz mapping in the
docstring is off by 2π. With `omega_m = 1.3` engine, the implicit
ω_m/(2π) is 1.3/(2π) ≈ 0.207 MHz, **not** 1.3 MHz as the docstring
asserts. To match Hasse's literal 1.3 MHz one should set
`omega_m = 2π · 1.3 ≈ 8.17`.

This affects no dimensionless plot in WP-V, but it does mean the
"absolute time" reading of ω_m·δt in §D1 above is itself ambiguous.
Either way, ω_m·δt is not zero — the rotating-frame approximation is
violated.

**Fix.** Pick one convention and enforce it everywhere. Recommend
angular (rad/μs) since it matches the code; update the docstring to say
"angular MHz, i.e. 2π·MHz_cyclic" and add the conversion factor in
`compute_derived`.

### D6 — N_pulses choice diverges from Hasse

Hasse: N_S = 30, δt = 100 ns, total = 23.1 μs. Engine default: N = 22,
δt derived from π/2 budget. The dimensionless ratio δt/T_m differs
between the two (~5% engine vs ~13% Hasse). Once D1 is fixed, this
ratio matters: Hasse's larger δt/T_m means *more* contrast loss per
pulse than the engine would predict even with motional H added.

**Fix.** Allow `delta_t_ns` to be set independently of `n_pulses`,
making the per-pulse rotation θ_pulse = Ω_eff · δt a derived quantity.
The sequence then becomes "N pulses of duration δt, separated by T_m,
total rotation ≈ N · θ_pulse" with no implicit π/2 budget. Hasse's
N=30, δt=100 ns gives total = N · Ω_eff · δt rad — a check that
Hasse's nominal parameters do produce ≈ π/2 (or document why they
don't, see Hasse Appendix D wording).

### D7 — LF mode tilt mapping (WP-V Fig 8 reproduction issue, not engine bug)

In `run_fig8.py` I used `η(tilt) = η_LF · cos(tilt)`, which is even in
tilt and gave identical curves at ±5°. Hasse Fig 8 caption shows
asymmetric green (−5°) and orange (+5°) curves, indicating the tilt
enters as `η = η_max · cos(θ_0 + tilt)` with the unperturbed AC-to-LF
angle θ_0 ≠ 0 (Hasse main text mentions a ≈ 30° geometry). To match
Hasse's tilt panel, the WP-V driver script needs the unperturbed angle
as a parameter, not the engine.

## 5. Recommended fix order (smallest → largest)

1. **D5** (units docstring): no code change, no behaviour change.
   Documentation only. Removes ambiguity from every other discussion.
2. **D3** (MW π/2 sync): one-line default change or expose a flag. Makes
   the engine's σ_z directly comparable to Hasse Fig 6a without basis
   rotation on the readout side. Affects every plot's mean σ_z baseline.
3. **D4** (coupling-block convention): null-test fix or comment.
4. **D2** (per-pulse AC phase): expose `ac_phase_deg`. Restores Fig 6b
   ϕ-dependence. Engine modification on the order of a dozen lines.
5. **D1** (motional H during pulse): structurally larger — adds an
   `intra_pulse_motion=True` mode that puts `ω_m a†a ⊗ I_2` into the
   per-pulse H. Restores Fig 8b. Likely also tightens Fig 6b amplitude
   to Hasse's ±0.9.
6. **D6** (independent δt parameter): cleanest after D1; makes the
   engine quantitatively comparable to Hasse's N=30, δt=100 ns timing.

Steps 1–4 are independent quick wins. Steps 5–6 are the structural
extension that turns the engine from a "stroboscopic-frame reproduction"
into a true "Hasse-protocol reproduction".

## 7. Status

WP-V findings (Fig 6b half-amplitude + missing ϕ-fringe; Fig 8b
flat C) are now traced to **two** specific engine simplifications (D1
and D2), not to user-side errors in the reproduction scripts. The
WP-V `run_fig6.py` and `run_fig8.py` drivers correctly query the engine
for what it can return; what they cannot return is what the engine
omits.

Recommended next action: implement D2 first (cheaper, restores Fig 6b
qualitative structure even without D1), then D1 (restores Fig 8b
quantitatively). Re-run WP-V after each fix to verify panel-by-panel
recovery. Each fix should land in a separate commit so the engine's
behaviour history stays auditable.

to the user.
